# Procedure Topic: Onboarding
## Fast Time-to-First-Contribution (TTFC) Procedure

### Problem Statement
Across HAAG research groups, onboarding for both researchers and computational advisors is inconsistent and often slower than necessary, being heavily dependent on informal Slack communication or waiting for clear structure during biweekly meetings. While weekly logs are intended to create accountability, they are written and reviewed inconsistently and primarily capture self-reported activity rather than a consistent and HAAG-verifiable contribution. This results in stalled early-semester project momentum, unclear expectations on deliverables, and fragmented contribution visibility, which can demotivate researchers as the semester goes on. Without a standardized requirement to ensure a new researcher produces a visible artifact within a given onboarding window, we see a structural gap between onboarding and contribution, which can be a big blow to both new projects and existing but potentially stagnated projects that need an injection of momentum and wins.

### Proposed Idea
In many software engineering teams, time-to-first-commit (or time-to-first-contribution) (TTFC) is one heuristic that effectively serves as a proxy for onboarding success. Anecdotally, it can help set up the structure for continued engagement, reducing initial cognitive overload, clarify early feedback loops, reinforce a team's best experimental and coding practices, and more.

The proposed idea is thus to have every researcher produce a visible artifact that is meaningful to the project--which can be in the form of a git commit, introducing a novel piece of literature or idea for review, or some other artifact that a project expects to produce throughout a given term, but it should be produced within a realistically short timeframe (which can be assessed after benchmarking current state), and it should not be trivial. This guidelines for the artifact should be defined by the faculty/computational advisors prior to researchers starting for the semester, and the computation advisor(s) would be responsible for validating the artifact within another TBD timeframe shortly after it is produced.

This procedure is intended to shift the focus on self-reported effort via the weekly logs (with respect to onboarding) in favor of visible artifacts that demonstrate meaningful contribution that can be reviewed with output-based accountability and can be iterated on, rather than being stuck in an ambiguous or perpetual analysis loop.

### Benchmarking Current State
To formalize the current state of onboarding, we would need data on the following, which should be able to be collected from the PMs:
- Current onboarding procedures by project
- Onboarding timeline by project, including:
    - When each researcher received access to resources such as repo, HPC clusters
    - Date of first contribution by researcher (TTFC) (whether that was providing a new piece of literature, git commit, running the software, etc.)
    - Date of first computation advisor validation
- Project timeline and goals

Collecting this information would assess how projects are currently doing with respect to onboarding, if there are trends that appear beneficial vs detrimental, see what bottlenecks can be identified, and try to assess any correlation between onboarding and downstream alignment between timeline and goals.

### Validation of Procedure Correctness
To validate whether this procedure achieves the intended outcomes, we can compare the metrics collected above, focusing on TTFC, how long it took computation advisor to validate, and if goals are being hit consistently.

### Intended Audience for Procedure

This procedure is intended for HAAG admins and faculty/computational advisors to monitor researchers' participation/onboarding experience, and require active researcher contribution.
